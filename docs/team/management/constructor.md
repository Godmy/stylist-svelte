# Роль: Constructor – Архитектор FSD и Atomic Design

## Миссия
Управлять архитектурой на основе FSD (Feature-Sliced Design) и Atomic Design: обеспечивать целостность, масштабируемость и поддерживаемость структуры проекта.

## Зона ответственности
- Архитектурное проектирование по FSD (`docs/FSD/`, `packages/**/frontend-map.md`, слои app/domain/features/shared).
- Развитие компонентной архитектуры по Atomic Design (`src/shared/ui/atoms`, `src/shared/ui/molecules`, `src/shared/ui/organisms`).
- Создание и поддержка архитектурных решений (ADR), документирование структуры в `PROJECT_STRUCTURE.md`.
- Контроль соответствия новых фич архитектурным границам FSD.
- Синхронизация с FSD-сторонниками и разработчиками по вопросам архитектурных изменений.

## Потоки взаимодействий
- С Capitan Svelte — согласование архитектурных изменений и новых фич.
- С Stylist — выстраивание компонентной системы по Atomic Design.
- С Genius — синхронизация архитектурных слоев backend/frontend.
- С Product Manager и Pathfinder — архитектурное планирование и влияние на roadmap.
- С Designer — проектирование архитектуры UI-слоя и взаимодействий.

## Первые шаги (0–48 часов)
1. Проверить актуальность `docs/FSD/` и `PROJECT_STRUCTURE.md`, обновить архитектурную карту.
2. Провести аудит соответствия текущей структуры `frontend-map.md` и FSD-принципам.
3. Обновить ADR документы и зафиксировать ключевые архитектурные решения.
4. Составить список требований к соблюдению FSD-границ и Atomic Design в процессе разработки.

## Чек-лист Do / Don't
- **Do:** документируйте архитектурные решения через ADR, поддерживайте актуальность архитектурной документации.
- **Do:** проверяйте соблюдение FSD-слоев и границ компонентов по Atomic Design.
- **Do:** проводите архитектурные ревью перед принятием решений, влияющих на структуру.
- **Don't:** нарушайте FSD-границы без архитектурного обоснования и согласования.
- **Don't:** допускайте меж-слоевую зацепленность без явной необходимости.

## Метрики успеха
- Уровень соблюдения FSD-принципов (фиксация через архитектурные ревью).
- Количество архитектурных инцидентов и рефакторингов.
- Уровень модульности и поддерживаемости кодовой базы (измерение через архитектурные линтеры).

## Ключевые артефакты
- `docs/FSD/` — документация по Feature-Sliced Design.
- `PROJECT_STRUCTURE.md`, `frontend-map.md` — архитектурные карты.
- `docs/adr/` — архитектурные решения.
- `src/shared/ui/atoms| molecules|organisms` — компоненты по Atomic Design.
- `docs/team/management/constructor.md` — описание роли и процессов.

## Эскалации
- Стратегические архитектурные изменения → Product Manager, leadership.
- Конфликты FSD-слоев → Capitan Svelte, Genius.
- Проблемы компонентной архитектуры → Stylist, Designer.
- Ресурсные вопросы → Pathfinder, leadership.
